# 배포 페이지 디자인 개선 PRD

## 1. 개요

| 항목 | 내용 |
|------|------|
| 대상 | audio-hotkeys 배포 페이지 (`site/index.html`, GitHub Pages) |
| 라이브 | https://kuroicode-beep.github.io/audio-hotkeys/ |
| 한 줄 요약 | 현재 텍스트 중심의 단일 파일 랜딩을 **첫인상·신뢰·공유성** 축으로 개선 |
| 문서 버전 | 1.0 |
| 작성일 | 2026-07-20 |
| 작성자 | Claude Code |
| 관련 스킬 | `svil-landing-page`(페이지 정본) · `svil-donation-section`(후원) · `svil-frontend-design`(디자인 정본) |

## 2. 배경 / 문제

현재 배포 페이지는 SVIL 고대비 다크 + 5개 언어 i18n + 토스 QR 후원까지 갖춘 **자체 완결형 단일 파일**로, 구조와 접근성 기반은 탄탄하다. 다만 방문자 관점에서 세 가지가 약하다.

1. **첫인상**: 히어로가 텍스트뿐이라, "이게 화면에서 어떻게 동작하는지"가 스크롤 전에 안 보인다. 트레이 앱이라 시각적 증거(OSD·트레이 메뉴)가 설득력의 핵심인데 그게 없다.
2. **신뢰**: 다운로드가 `.exe` 직링크인데 파일 크기·최신 버전·SmartScreen 경고 대처 안내가 없어, 저시력·비개발자 사용자는 "이거 눌러도 되나" 망설인다.
3. **공유성**: `og:`/`twitter:` 메타가 없어 카톡·SNS·블로그에 링크를 붙이면 **미리보기 카드가 안 뜬다**. 배포 페이지의 확산 경로를 스스로 막고 있다.

## 3. 목표 / 비목표

### 3.1 목표

1. 스크롤 전(above-the-fold)에 **제품 실제 화면**을 보여 3초 안에 "무엇인지" 전달.
2. 다운로드 결정을 돕는 **정보·안심 요소**(크기·버전·실행 안내)를 붙인다.
3. 링크 공유 시 **미리보기 카드**가 뜨게 한다(OG/Twitter 메타 + 대표 이미지).
4. 기능 카드의 **스캔성**을 높인다(색만이 아닌 아이콘+텍스트 라벨).
5. 개선을 `svil-landing-page` 스킬 템플릿에 **역반영**해 이후 모든 SVIL 페이지가 상속.

### 3.2 비목표 (이번 개선에서 하지 않음)

- 멀티 페이지·라우팅, 정적 사이트 생성기(SSG) 도입
- 프레임워크·번들러·빌드 툴체인 (단일 파일·무의존 유지)
- 외부 CDN·웹폰트·애널리틱스 스크립트 (SVIL 규칙: 자체 완결형)
- 라이트 테마 (SVIL 다크 강제 유지)
- 백엔드·폼 제출·뉴스레터

## 4. 현재 상태 진단

| 영역 | 현재 | 문제 |
|------|------|------|
| 히어로 | 제목·태그라인·CTA·메타(텍스트) | 제품 이미지 0. 무엇인지 감이 늦다 |
| 스크린샷 | 없음 | OSD·트레이·설정 등 핵심 UX가 안 보임 |
| 기능 카드 | 6개, 텍스트만 | 훑어보기 어려움. 시각 앵커 없음 |
| 다운로드 | `.exe` 직링크 | 크기·버전·서명/SmartScreen 안내 없음 |
| 공유 메타 | `description`만 | OG/Twitter 없음 → 링크 미리보기 안 뜸 |
| 버전 | 헤더 배지(v1.4.4) | "무엇이 새로운지" 진입점 없음 |
| 후원 | 토스 QR (신규) | 양호 |
| 접근성 | 포커스 링·reduced-motion·keep-all·lang 전환 | 기반은 OK. skip-link·heading 순서·대비 재감사 필요 |

## 5. 개선 요구사항

우선순위: **P0**(핵심, 이번에 반드시) · **P1**(권장) · **P2**(여력 시).

### P0-1. 히어로 제품 비주얼

- 히어로에 **실제 화면 이미지**(프로필 OSD 또는 트레이 메뉴)를 넣는다. 다크 배경과 어울리는 프레임/그림자로 배치.
- 이미지는 로컬(`assets/`) 최적화 PNG/WebP, `loading` 즉시(히어로는 lazy 금지), `width`/`height` 명시로 레이아웃 이동(CLS) 방지, `alt` 필수.
- **수용 기준**: 데스크톱·모바일 모두 스크롤 없이 제품 화면이 보인다. 이미지 누락 시에도 텍스트 히어로가 깨지지 않는다.

### P0-2. 공유 메타 (OG / Twitter)

- `og:title`·`og:description`·`og:image`·`og:url`·`og:type`, `twitter:card=summary_large_image` 추가. `og:image`는 1200×630 대표 이미지(`assets/og.png`), 절대 URL.
- **수용 기준**: 카톡·트위터·디스코드·블로그에 링크를 붙였을 때 **제목+설명+이미지 카드**가 렌더된다(공유 디버거로 확인).

### P0-3. 다운로드 안심 요소

- 다운로드 버튼 주변에 **파일 크기·대상 OS·최신 버전** 표기(버전은 모노체). "무료·오픈소스·MIT" 유지.
- **SmartScreen/서명 안내**: 서명 안 된 exe라 Windows가 경고할 수 있음을 1줄로 안내(접힘/툴팁 가능). 색만이 아니라 텍스트 라벨로.
- **수용 기준**: 사용자가 클릭 전 크기·버전·"경고 뜨면 어떻게"를 알 수 있다.

### P1-1. 스크린샷 섹션

- OSD·트레이 메뉴·설정 창을 보여주는 갤러리 섹션. 각 이미지에 짧은 캡션(i18n). 다크 프레임 통일.
- **수용 기준**: 핵심 화면 3종 이상이 캡션과 함께 보인다. 모바일에서 세로로 자연스럽게 쌓인다.

### P1-2. 기능 카드 아이콘

- 6개 카드에 **단색 글리프(인라인 SVG)** + 기존 텍스트 제목 병행. 색만으로 의미를 주지 않는다(SVIL 규칙).
- **수용 기준**: 아이콘 없이도(대체 텍스트/제목으로) 의미가 전달된다. 대비 ≥ 3:1.

### P1-3. "무엇이 새로운지" 진입점

- 헤더 버전 배지 또는 히어로 근처에 최신 버전 요약 + 업데이트 히스토리(앱 `VERSION_HISTORY` 또는 릴리스)로의 링크.
- **수용 기준**: 최신 버전과 변경 요약에 한 번에 닿는다.

### P2-1. 시각 리듬·깊이

- 섹션 간 배경 톤 교차(`--bg`/`--surface`)나 미묘한 구분선/그라디언트로 리듬. **애니메이션은 `prefers-reduced-motion` 존중**, 과한 모션 금지.
- **수용 기준**: 위계가 크기·색·여백으로만 표현되고(bold 합성 금지), reduced-motion에서 정적으로 동작.

## 6. 접근성 요구사항 (SVIL, 예외 없음)

- 본문 최소 12px 이상, 위계는 크기·색만(단일 굵기 폰트 — **bold 합성 금지**).
- 본문 대비 ≥ 4.5:1, UI ≥ 3:1. **모든 토큰 대비 재감사**(추가하는 이미지 위 텍스트 포함).
- **색상만으로 상태·의미 구분 금지** — 텍스트 라벨 병행.
- 터치 타겟: 본문 링크/버튼 ≥ 44–56px 유지.
- **skip-link**(본문 바로가기) 추가, heading 순서 h1→h2→h3 검증, 모든 이미지 `alt`, 언어 전환 시 `lang` 갱신 유지.
- `prefers-reduced-motion` + (가능하면) `prefers-contrast` 대응.

## 7. 디자인 원칙 / 제약

- **자체 완결형 단일 파일 유지.** 이미지도 `site/assets/` 로컬. 외부 요청 0.
- **색은 `theme` 토큰(CSS 변수)만.** 하드코딩 금지.
- **다국어 5종 동시.** 새 문구는 ko/en/ja/zh/vi 전부, 영어 디폴트 폴백. 키 개수 일치.
- 이미지 총량이 커지면 WebP·적정 해상도로 관리(단일 파일 원칙은 HTML 기준, 이미지는 `assets/` 허용).

## 8. 성공 지표

측정 도구가 가벼운 솔로 프로젝트임을 감안한 실용 지표:

- **Lighthouse 접근성 ≥ 95**, 성능 ≥ 90(모바일).
- 링크 공유 미리보기 카드 **정상 렌더**(카톡/트위터/디스코드 3곳 확인).
- 히어로에서 **첫 의미 있는 시각요소(제품 화면)**가 above-the-fold.
- 외부 네트워크 요청 **0건**(자체 완결성 회귀 없음, DevTools Network로 확인).

## 10. 리스크 / 유의

- **이미지 무게 vs 자체 완결성**: 스크린샷·OG 이미지가 커지면 로딩 저하. WebP·해상도 관리로 상쇄.
- **SmartScreen 안내 톤**: 불안을 키우지 않게 사실 위주 1줄로. 색이 아닌 라벨.
- **후원 QR 금융 정보**: 페이지 개편 중에도 QR/계좌 노출 원칙(이미지만·후원 전용 계좌) 유지.
- **스킬 드리프트**: M3에서 반드시 `svil-landing-page` 템플릿에 반영해야 다음 페이지가 개선을 상속.
